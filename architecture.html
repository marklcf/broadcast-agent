①这个项目在做什么

一句话：把「读一堆信息源 → 总结 → 每天定时推送给你」这件事,做成无人值守的 agent。

它真正的教学目标不是「抓资讯」,而是借这个场景演示 当上下文（context）会爆炸时,工程上怎么省 token、省注意力。整个代码库刻意保留了一个「反面教材」开关 --naive,让你亲眼量出好坏两种写法的 input_tokens 差距。

📡抓取

从 3 个数据源拉取当天的轻量条目（标题/链接/元数据,不抓正文）。

🧪隔离摘要

每个源派一个子 agent,在干净窗口里读条目、只回传一段精炼摘要。

📝去重 & 记忆

把历史播报卸载到 digest.md 文件,明天读回来去重。

📤投递

按配置选 Gmail / 飞书 / 本地 三种渠道之一推送。

③四个核心概念（这才是重点）

整个代码库就是为了把这 4 个 Context Engineering 手法变成你能跑、能量化的东西。
① 按需检索 · RETRIEVAL ON DEMAND

先只抓轻量信号,正文按需才取。gather() 只拿标题/链接/摘要元数据,绝不一上来就把每个链接的整页 HTML、整篇 PDF 拉进来。正文是「重」的,只有真正需要时才取。

② 隔离 · ISOLATION

每个数据源派一个子 agent,在独立上下文里干活。子 agent 读条目烧掉的 token 留在它自己的调用里,主流程只拿回一段摘要。这样主上下文永远小,且各源互不污染。

③ 卸载 · OFFLOAD

agent 的「记性」不在模型里,在 digest.md 这个文件里。每天的播报写到上下文窗口之外的文件,明天读回来续上。删了这个文件,agent 就失忆 —— 记忆在文件,不在模型。

④ 压缩 · COMPACTION

历史不全量带进上下文,只回最近一小段。history_compacted() 只取文件末尾 ~800 字符,避免上下文随天数线性膨胀。去重也只比对 URL 指纹,不是把整个历史塞回模型。

④模块逐个详解

def gather() -> dict:

🌐 三个数据源 — 真实工程的三种取数姿态

源文件取数方式工程教学点
arXivsources/arxiv.py官方 Atom API + feedparser复用 Lab-1 的 429 纪律:共享 client、自定义 UA、间隔 ≥3 秒、撞 429 直接抛不盲目重试
Hacker Newssources/hackernews.pyAlgolia 搜索 API（无 auth）用 search_by_date + points>N 过滤低分噪声,一次请求带回标题/分数/评论数
GitHub Trendingsources/github_trending.py爬 HTML（BeautifulSoup）没有官方 API 是常态:爬 HTML 拿「今日新增 star」速度信号,但选择器会随改版失效

三个源都导出统一接口 fetch(cfg) → [轻量条目],每条形如 {source, title, url, meta} —— 这种统一契约让主流程能无差别地遍历它们。

⑤正确版 vs 反面教材

跑 python run_broadcast.py --naive 看 token 爆炸,这是整堂课最直观的一刀。
--naive（反面教材）正常跑（正确版）
取数连每个链接的整页正文都抓回来只取轻量条目（标题/链接/元数据）
上下文所有源的全文塞进一个主上下文每源子 agent 隔离,主流程只收摘要
主窗口 input_tokens爆炸 💥小很多 ✓
体感又慢、又贵、又糊快、省、信噪比高
为什么这样设计

反面教材不是 bug,是刻意保留的对照实验。两次运行打印的 input_tokens 差距,就是「隔离 + 按需检索」省下来的真金白银。这比任何 PPT 都有说服力。

🚀GitHub Actions
deploy/daily.yml

schedule: cron "17 1 * * *"（避开整点高峰）让 GitHub 每天免费替你跑、关机照跑。密钥放仓库 Secrets,跑完把 digest.md 提交回仓库。

kp-04 · 渠道选择与「凭据即密码」

Gmail vs 飞书的选择取决于「你在哪发、你在哪读」:GFW 只挡「从大陆读 Gmail」,不挡「从境外发 Gmail」。飞书 webhook 的 URL 本身就是凭据,谁拿到谁能发 —— 所有密钥只进 .env / Secrets,绝不进 git。
