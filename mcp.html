MCP Model Context Protocol

这一页只讲一件事:在这个 lab 里,「用别人的 MCP」到底是怎么接上、发现、调用、收尾的 —— 从一段 config 到 agent 真去碰 GitHub 的完整链路。

MCP 是什么 三层架构 配置:.mcp.json 发现:tools/list 三积木 调用:tools/call 完整流程 P5:用 MCP 搭自动化 MCP 的边界

①MCP 是什么 · 在本 lab 的角色

一句话:MCP 是「agent ↔ 工具」之间的标准协议。你消费别人发布好的工具,而不实现它。

本 lab 不让你写一行 GitHub API 代码。你只在 .mcp.json 里加一段 config,Claude Code 就能凭空多出一排 GitHub 工具(查 repo、读文件、列 issue、看 PR、搜代码……)。这些工具的实现全在别人发布的 server 进程里 —— 官方的 @modelcontextprotocol/server-github。

用 vs 造

本 lab 的 MCP 出现在两个阶段:P1 用它读 GitHub 数据(只读 token)、P5 用它建仓+推 workflow 文件搭自动化(写 token)。同一个 GitHub MCP,两次不同用途。这节课只「用」别人的;「造」自己的 MCP server 是后面的课。

②三层架构 · Host / Client / Server

对应 slides 第 7 幕。这是理解后面一切调用的地图。
HOST
Claude Code
你的 agent 运行环境,模型在这里思考、决定调哪个工具
CLIENT
那条连接
Host 内部维护的、连到 server 的协议连接;负责 tools/list 发现、tools/call 转发
SERVER
github 进程
npx 启动的 server-github;真正持有 GitHub API 实现 + 你的 token
↑ JSON-RPC over stdio(标准输入输出管道) ↑
github 进程 ⟷ 真实 GitHub REST API
关键直觉

模型能看到的只有工具的「描述 + input schema」,看不到任何实现。它先靠 tools/list 发现有哪些工具,再用 tools/call 请求执行 —— 真正干活的是 server 进程,模型只是「点单」。

③配置 · .mcp.json 逐字段

整个「接上别人 MCP」就靠这 11 行 —— 没有一行 GitHub API 实现代码。
{
  "mcpServers": {
    "github": {                                          // ← 工具前缀,/mcp 里看到的就叫 github
      "command": "npx",                                  // 用 npx 启动(不必预装)
      "args": ["-y", "@modelcontextprotocol/server-github"], // 别人发布好的 server 包
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PERSONAL_ACCESS_TOKEN}"  // 从环境注入,不写死
      }
    }
  }
}
字段作用
"github"server 的名字 = 工具命名空间前缀。Claude Code 里 /mcp 看到的一排工具都挂在它下面。
command + args怎么启动这个 server。npx -y 表示按需下载并运行,无需手动 install。
env传给 server 进程的环境变量。${...} 语法从你的 shell / .env 注入 token,凭据不进 git。

④发现 · /mcp 与 tools/list

工具是怎么「凭空出现」的?Client 在背后发了一次 tools/list。

在 Claude Code 里输入 /mcp,你会看到 github 那排工具。想看 server 内部到底长什么样(每个工具的描述、input schema),用官方 MCP Inspector 的 CLI 模式直接 dump:

# 先让 token 进环境
export GITHUB_PERSONAL_ACCESS_TOKEN=<你的-token>

# dump server 暴露的全部工具:名字 + 描述 + inputSchema
npx @modelcontextprotocol/inspector --cli \
  -e GITHUB_PERSONAL_ACCESS_TOKEN=$GITHUB_PERSONAL_ACCESS_TOKEN \
  npx -y @modelcontextprotocol/server-github \
  --method tools/list
这正是模型 client 背后做的事

打印出的 JSON 里,每个工具就是 name + description + inputSchema。Client 先 tools/list 发现工具,把这些描述喂给模型;模型据此决定调谁、传什么参数 —— 它看得到 schema,看不到实现。Inspector 让你绕过模型、自己当 client,看清协议本身。

⑤三积木 · Tools / Resources / Prompts

一个 MCP server 可以暴露三种 primitive。本 lab 主要用 tools。
🔧
Tools
能「做」的动作。tools/list 发现、tools/call 执行。如 search_repositories、list_issues。
📄
Resources
能「读」的资源。resources/list 列出。如某个文件、某条数据的只读快照。
💬
Prompts
能复用的提示词模板。prompts/list 列出。预置好的、可参数化的对话起点。
# 把 --method 换掉,各看一眼另两种 primitive
  --method resources/list   # 能「读」的资源
  --method prompts/list     # 能复用的提示词模板

⑥调用 · tools/call

你可以让模型来调,也可以自己当 client 直接调(绕过模型)。

方式 A · 自己当 client,直接调(Inspector)

# tools/call:指定工具名 + 参数,不经过模型
npx @modelcontextprotocol/inspector --cli \
  -e GITHUB_PERSONAL_ACCESS_TOKEN=$GITHUB_PERSONAL_ACCESS_TOKEN \
  npx -y @modelcontextprotocol/server-github \
  --method tools/call \
  --tool-name search_repositories \
  --tool-arg query=modelcontextprotocol

方式 B · 指挥 agent 用(自然语言,本 lab 的主线)

不写代码,用自然语言指挥,并明确要求走 MCP 工具:

P1 的指挥语

「用 GitHub MCP 工具(不要用 gh),看一下 anthropics/anthropic-sdk-python 的几条 open issue + 最近几个 commit,挑出对我们这门课有用的。」

盯着 agent 调的是 list_issues / list_commits / get_file_contents 这些 MCP 工具,拿回真实 GitHub 数据。

⚠️ 常见陷阱:agent 偷用 gh

你机器里有 gh CLI,agent 默认爱用它。一旦它开始敲 gh,拦住:「走 GitHub MCP 工具,我要看 MCP 在干活。」—— 这节课的魂就是看见 MCP 替你做事,用 gh 就白上了。

⑦完整调用流程 · 从一句话到 GitHub 数据

把前面拆开的环节串起来 —— 一次「用 MCP 读 issue」背后发生了什么。
启动 server
Claude Code 读 .mcp.json,按 command/args 用 npx 拉起 github server 进程,把 token 经 env 注入。
npx -y @modelcontextprotocol/server-github
发现工具
Client 通过 stdio 向 server 发 tools/list,拿回所有工具的 name + description + inputSchema。
Client → Server: tools/list ⟶ [search_repositories, list_issues, ...]
模型决策
你用自然语言提需求。模型只看工具描述 + schema,决定「该调 list_issues,参数 owner/repo」。
model: 「我要用 list_issues(owner=anthropics, repo=...)」
转发调用
Client 把模型的选择封成 tools/call 请求发给 server。
Client → Server: tools/call { name:"list_issues", arguments:{...} }
server 干活
server 进程用 token 调真实 GitHub REST API,拿回 JSON。
github 进程 → GitHub API → 真实 issue 数据
结果回灌
结果经 Client 回到模型上下文,模型据此继续推理、给你最终答案。
Server → Client → model context → 你看到的回答

⑧P5 · 同一个 MCP,这次用来搭自动化

P1 用 MCP「读」,P5 用 MCP「写」——建仓 + 推 workflow 文件。
P5 的指挥语(只用写权限工具)

「用 GitHub MCP 工具(create_repository / push_files),新建一个 private 仓库 my-daily-broadcast,把 broadcast 代码推上去(绝不 push .env / .venv),再把 deploy/daily.yml 推到 .github/workflows/daily.yml。全程只用 MCP 工具,不要 gh、不要 git push。」

阶段用到的 MCP 工具需要的 token 权限
P1 读数据list_issues list_commits get_file_contents search_repositories只读 · fine-grained,只勾 public_repo 读
P5 搭自动化create_repository push_files / create_or_update_file写 · classic PAT,勾 repo + workflow
kp-04 活教材:最小权限,按需升级

P1 给只读、P5 才升到写。权限按需给、当 secret 管、用完可删。GitHub MCP 用的就是 .mcp.json 里那个 GITHUB_PERSONAL_ACCESS_TOKEN,P5 时把它换成有写权限的。

⑨MCP 的边界 · 它做不到什么

诚实地用工具:看清 MCP 能干什么、不能干什么。
边界 1 · MCP 没有时钟(kp-05)

能不能让 GitHub MCP 自己每天定时跑?—— 不能。MCP server 是「手」,被调用才动。「每天定时」必须交给外部调度器(本 lab 是 GitHub Actions cron)。三层:Actions cron(WHEN) → cron_entry.py(WHAT) → MCP/API(碰世界的手)。别把 GitHub MCP(手)和 GitHub Actions(闹钟+免费算力)搞混。

边界 2 · MCP 不能设 secret

GitHub MCP 没有设 secret 的工具(secret 要 libsodium 加密)。所以配 Actions Secrets 这一步只能退回网页 / gh secret set —— 而且 agent 要明说「这步 MCP 没有工具,所以用 gh」。能建仓、能推文件(MCP)≠ 能设密(MCP 不行)。

✅ 学完这页你应该能说出

  • 用别人的 MCP = 加一段 config,Client 靠 tools/list 自动发现工具,你没写一行工具实现。
  • 模型只看到工具的描述 + input schema,看不到实现;调用走 tools/call。
  • 三层:Host(Claude Code)/ Client(连接)/ Server(github 进程);三积木:tools / resources / prompts。
  • MCP 没有时钟、不能设密 —— 调度交给 Actions,设密退回 gh。这才是诚实地用工具。
